# Render 워크플로우

Render 워크플로우는 서비스를 배포할 때마다 실행되는 전체 단계 시퀀스입니다 — 변경이 트리거된 순간부터 트래픽이 새 버전으로 전환되는 순간까지. 각 단계를 이해하면 더 빠르게 배포하고, 문제를 더 일찍 발견하고, 장애에서 자신 있게 복구할 수 있습니다.

## 배포가 트리거되는 방법

모든 배포는 트리거로 시작됩니다. Render는 세 가지를 지원합니다:

| 트리거 | 실행 시기 |
|---|---|
| **자동 배포** | 설정된 브랜치에 모든 푸시에서 자동으로 (기본적으로 활성화됨). |
| **수동 배포** | 대시보드(**수동 배포 → 최신 커밋 배포**) 또는 Render API를 통해 요청 시. |
| **배포 훅** | 서비스 설정에서 생성한 인증된 웹훅 URL을 통해 요청 시. |

**설정 → 빌드 & 배포 → 자동 배포**에서 자동 배포를 설정합니다. 이를 끄면 자체 CI 파이프라인이나 승인 단계 뒤에 배포를 차단할 수 있습니다 — 검사가 통과된 후에만 배포 훅이나 API를 호출합니다.

### 배포 훅

배포 훅은 고유한 HTTPS URL입니다. 해당 URL에 `POST` 요청을 보내면 설정된 브랜치의 최신 커밋 배포가 시작됩니다. 훅을 사용하면 전체 대시보드 액세스를 부여하지 않고도 모든 CI 시스템, 스크립트, 또는 외부 서비스에 Render 배포를 통합할 수 있습니다.

**설정 → 빌드 & 배포 → 배포 훅**에서 훅을 생성합니다. URL을 비밀로 취급하세요 — URL을 가진 누구나 배포를 트리거할 수 있습니다.

## 빌드 단계

배포가 트리거된 후 Render는 새 빌드 컨테이너 내에서 **빌드 명령**을 실행합니다. 컨테이너는 배포별로 격리됩니다. 이전 빌드의 파일시스템에서 아무것도 이어지지 않습니다 (모듈 및 패키지 캐시는 별도로 저장됩니다 — 아래 [캐싱](#캐싱) 참조).

Render는 빌드 출력을 대시보드 로그 뷰어에 실시간으로 스트리밍합니다. 빌드 명령에서 0이 아닌 종료 코드는 배포를 즉시 실패시킵니다. 현재 버전은 계속 트래픽을 처리합니다.

### 런타임별 기본 빌드 명령

| 런타임 | 기본 빌드 명령 |
|---|---|
| Go | `go build -o server .` |
| Node | `npm install; npm run build` |
| Python | `pip install -r requirements.txt` |
| Ruby | `bundle install` |
| Docker | _(Dockerfile에서 이미지 빌드)_ |

**설정 → 빌드 & 배포 → 빌드 명령**에서 모든 기본값을 재정의합니다.

### 캐싱

Render는 빌드 간에 언어 패키지 매니저를 캐시합니다 (Go 모듈 캐시, `node_modules`, pip 휠 캐시 등). 캐시는 서비스별, 브랜치별로 키가 지정됩니다. 의존성을 추가하거나 제거하는 커밋은 변경된 패키지만 무효화합니다 — 나머지 캐시는 따뜻하게 유지됩니다.

강제로 클린 빌드를 수행하려면 대시보드에서 **수동 배포 → 빌드 캐시 지우기 및 배포**를 선택합니다.

## 사전 배포 명령

**사전 배포 명령**은 빌드가 성공한 *후* 트래픽이 새 버전으로 전환되기 *전*에 수명이 짧은 컨테이너 내에서 실행됩니다. 이 단계를 배포와 원자적이어야 하는 작업에 사용합니다:

- 데이터베이스 마이그레이션
- 캐시 워밍
- 스키마 유효성 검사

**설정 → 빌드 & 배포 → 사전 배포 명령**에서 명령을 설정합니다. 명령이 0이 아닌 값으로 종료되면 Render는 배포를 취소하고 이전 버전을 그대로 유지합니다. 트래픽은 중단되지 않습니다.

> **사전 배포 명령을 멱등성 있게 유지하세요.** 부분 실패 후 배포가 재시도되면 사전 배포 명령이 다시 실행됩니다. 마이그레이션 및 기타 상태 변경 작업은 두 번 이상 실행해도 안전해야 합니다.

## 헬스 체크

새 인스턴스가 시작되면 Render는 트래픽을 라우팅하기 전에 헬스 체크를 통과하기를 기다립니다. 웹 서비스의 경우 Render는 설정된 **헬스 체크 경로**(기본값: `/`)에 HTTP GET을 보냅니다. 서비스는 타임아웃 창 내에 `2xx` 응답을 반환해야 합니다.

**설정 → 헬스 & 알림 → 헬스 체크 경로**에서 경로를 설정합니다. 중요한 의존성(데이터베이스 연결, 필수 설정)을 확인하는 전용 엔드포인트(`/healthz` 또는 `/ping`)가 가장 신뢰할 수 있는 신호를 제공합니다.

헬스 체크가 성공하지 못하면:

- 배포가 실패로 표시됩니다.
- 이전 버전이 계속 트래픽을 처리합니다.
- 새 인스턴스가 중지됩니다.

## 배포 라이프사이클 한눈에 보기

```
트리거 수신
  └─ 빌드 명령 실행
       └─ [빌드 실패] → 배포 취소, 이전 버전 유지
       └─ 빌드 성공
            └─ 사전 배포 명령 실행 (설정된 경우)
                 └─ [사전 배포 실패] → 배포 취소, 이전 버전 유지
                 └─ 사전 배포 성공
                      └─ 새 인스턴스 시작, 헬스 체크 시작
                           └─ [헬스 체크 실패] → 배포 실패, 이전 버전 유지
                           └─ 헬스 체크 통과
                                └─ 트래픽이 새 버전으로 전환
                                     └─ 이전 인스턴스 드레인 및 중지
```

## 무중단 배포

Render는 유료 플랜에서 무중단 배포를 수행합니다. 이전 버전이 중지 신호를 받기 전에 새 버전이 헬스 체크를 통과해야 합니다. 겹치는 창 동안 두 인스턴스가 모두 실행 중입니다 — 두 버전이 안전하게 공존할 수 있도록 애플리케이션을 설계합니다:

- 데이터베이스 스키마 변경을 이전 버전의 코드와 하위 호환 가능하게 만듭니다.
- 새 스키마를 기대하는 코드를 추가하는 같은 배포에서 컬럼을 제거하거나 필드 이름을 변경하지 마세요.
- 사전 배포 명령을 사용하여 새 코드가 트래픽을 처리하기 *전에* 마이그레이션을 실행합니다.

## 롤백

이전 배포로 롤백하려면 대시보드에서 서비스를 열고 **이벤트** 탭으로 이동하여 복원하려는 배포를 찾고 **이 배포로 롤백**을 선택합니다. Render는 롤백된 버전에 대해 사전 배포 명령을 포함한 동일한 워크플로우를 실행합니다.

롤백은 사전 배포 명령을 재생하므로 마이그레이션이 가역적이거나 이전 버전의 코드로 롤백하는 것이 현재 데이터베이스 스키마에 대해 안전한지 확인합니다.

## 풀 리퀘스트 프리뷰

서비스에 대해 **풀 리퀘스트 프리뷰**를 활성화하면 Render가 각 열린 풀 리퀘스트에 대해 서비스의 임시 복사본을 자동으로 생성합니다. 프리뷰는 동일한 빌드 명령, 환경 변수(명시적으로 제외한 것 제외), 그리고 기본 서비스와 동일한 설정을 사용합니다.

프리뷰는 PR이 열릴 때 생성되고 닫히거나 병합될 때 삭제됩니다. 프리뷰 URL 링크가 풀 리퀘스트에 직접 게시됩니다.

**설정 → 풀 리퀘스트 프리뷰**에서 프리뷰를 활성화합니다. 모든 환경 변수를 상속하거나 프리뷰 환경을 위한 별도 세트를 설정할 수 있습니다.

> **프리뷰 환경은 리소스를 공유합니다.** 서비스가 데이터베이스에 연결하는 경우 프리뷰용 `DATABASE_URL`을 재정의하지 않으면 프리뷰가 동일한 데이터베이스에 연결됩니다. 격리된 프리뷰 테스트를 위해 별도의 데이터베이스나 스키마를 사용하세요.

## 알림 및 경보

Render는 배포가 성공, 실패 또는 롤백될 때 알릴 수 있습니다. **설정 → 알림**에서 알림 대상을 설정합니다:

- **이메일** — 서비스 소유자 또는 설정된 주소로 전송됩니다.
- **Slack** — 웹훅을 통해 채널에 배포 이벤트를 게시합니다.
- **PagerDuty / 일반 웹훅** — 모든 엔드포인트에 이벤트를 전달합니다.

헬스 체크 실패도 경보를 트리거합니다. **설정 → 헬스 & 알림**에서 경보 임계값을 설정합니다.

## 환경 프로모션 패턴

스테이징과 프로덕션 환경을 거치는 서비스의 경우 일반적인 패턴은 다음과 같습니다:

1. **스테이징 서비스** — `main`에 모든 푸시에서 자동 배포됩니다. 배포 훅은 비활성화됩니다.
2. **프로덕션 서비스** — 자동 배포가 꺼져 있습니다. 스테이징이 승인 기준을 통과한 후에만 배포 훅이 호출됩니다.

이렇게 하면 두 환경 모두에 동일한 Render 워크플로우 인프라를 사용하면서 스테이징과 프로덕션 사이에 사람이나 자동화된 게이트를 둘 수 있습니다.

## 일반적인 문제

**빌드는 성공하지만 배포가 활성화되지 않음**
헬스 체크가 실패하고 있을 가능성이 높습니다. 앱이 `PORT` 환경 변수에 바인드되어 있고 타임아웃 전에 설정된 헬스 체크 경로에서 `2xx`로 응답하는지 확인합니다.

**사전 배포 명령이 마이그레이션이 이미 적용된 경우에도 매번 실행됨**
이는 예상된 동작입니다 — 스키마가 이미 최신 상태일 때 no-op이 되도록 마이그레이션 도구를 설계합니다 (대부분의 마이그레이션 프레임워크는 기본적으로 이렇게 합니다).

**설정하지 않은 브랜치에서 자동 배포가 트리거됨**
**설정 → 빌드 & 배포 → 브랜치**에서 브랜치 이름을 확인합니다. 저장소에서 브랜치 이름 변경은 Render 설정을 자동으로 업데이트하지 않습니다.

**새 배포 후에도 이전 버전이 계속 트래픽을 처리함**
무료 티어 서비스에서 Render는 무중단 배포를 수행하지 않습니다. 이전 인스턴스가 중지되고 새 인스턴스가 시작되는 동안 어느 버전도 트래픽을 처리하지 않는 짧은 기간이 있습니다.
