# Render Go

Render는 Go를 기본적으로 지원합니다. 추가 툴링 없이도 Go 애플리케이션을 웹 서비스, 백그라운드 워커, 또는 크론 잡으로 배포할 수 있습니다. Render가 Go 모듈을 감지하고, 바이너리를 빌드하고, 실행합니다.

## 요구 사항

- 저장소 루트(또는 루트 디렉터리로 지정한 하위 디렉터리)에 `go.mod` 파일이 있어야 합니다.
- Go 1.20 이상을 권장합니다. Render는 최신 안정 릴리스를 기본값으로 사용합니다. 앱에 특정 버전이 필요한 경우 `GOVERSION` 환경 변수로 고정하세요.

## 빠른 시작

1. Go 저장소를 GitHub 또는 GitLab에 푸시합니다.
2. Render 대시보드에서 **새로 만들기 → 웹 서비스**를 선택하고 저장소를 연결합니다.
3. 감지된 빌드 및 시작 명령을 확인하거나 (아래 참조) 변경합니다.
4. **웹 서비스 생성**을 클릭합니다.

Render는 설정된 브랜치에 푸시할 때마다 서비스를 자동으로 빌드하고 배포합니다.

## 빌드 및 시작 명령

Render는 Go 프로젝트를 자동으로 감지하고 합리적인 기본값을 설정합니다.

| 설정 | 기본값 |
|---|---|
| **빌드 명령** | `go build -o server .` |
| **시작 명령** | `./server` |

`main` 패키지가 하위 디렉터리에 있는 경우 빌드 명령을 적절히 조정합니다:

```bash
# main 패키지가 cmd/api에 있는 프로젝트의 빌드 명령
go build -o server ./cmd/api
```

언제든지 서비스의 **설정 → 빌드 & 배포** 섹션에서 각 명령을 재정의할 수 있습니다.

## Go 버전 고정

서비스에서 `GOVERSION` 환경 변수를 설정하여 특정 릴리스를 요청합니다:

```
GOVERSION=1.22.4
```

Render는 빌드 시 요청한 버전을 설치합니다. 다음 배포 전에 변수를 설정해야 적용됩니다.

## 환경 변수

Render에서 실행 중인 Go 앱은 로컬과 동일한 방식으로 `os.Getenv`를 통해 환경 변수를 읽습니다. 서비스의 **환경** 탭에서 변수를 설정하거나 Render 환경 그룹을 참조하여 서비스 간에 값을 공유합니다.

앱은 어떤 포트에서 수신할지 알기 위해 `PORT` 환경 변수를 읽어야 합니다:

```go
port := os.Getenv("PORT")
if port == "" {
    port = "8080"
}
http.ListenAndServe(":"+port, nil)
```

Render는 웹 서비스에 대해 `PORT`를 자동으로 주입합니다. 포트 번호를 하드코딩하면 헬스 체크가 실패합니다.

## 헬스 체크

Render는 서비스가 실행 중인지 확인하기 위해 기본적으로 `/`에 HTTP GET 요청을 보냅니다. 앱이 전용 헬스 엔드포인트(예: `/healthz` 또는 `/ping`)를 노출하는 경우 **설정 → 헬스 & 알림 → 헬스 체크 경로**에서 다른 경로를 설정합니다.

체크는 기본 타임아웃 내에 2xx 응답을 기대합니다. 시간 내에 `PORT`에 바인드하지 않는 서비스는 비정상으로 표시되고 배포가 롤백됩니다.

## 정적 파일 및 에셋

앱이 정적 에셋을 제공하는 경우 `//go:embed`로 임베드하거나 작업 디렉터리에 상대적으로 포함합니다. 런타임의 작업 디렉터리는 저장소의 루트입니다. 절대적인 빌드 머신 경로에 의존하는 경로는 올바르게 해석되지 않습니다.

## 비공개 모듈

비공개 저장소에서 모듈을 가져오려면 `GONOSUMCHECK` 또는 `GONOSUMDB` 변수를 추가하고 `NETRC` 환경 변수나 시크릿 파일을 통해 주입된 `~/.netrc` 파일을 통해 자격 증명을 제공합니다. Render는 빌드 컨테이너 외부에서 비공개 모듈 다운로드를 캐시하지 않으므로 각 배포는 새로운 복사본을 가져옵니다.

## 모노레포 설정

서비스가 모노레포의 여러 서비스 중 하나인 경우 서비스의 **루트 디렉터리**를 `go.mod`가 포함된 하위 디렉터리로 설정합니다. 그러면 Render가 해당 디렉터리를 기준으로 모든 빌드 및 시작 명령을 실행합니다.

## 크론 잡 및 백그라운드 워커

Go 바이너리는 Render에서 백그라운드 워커 또는 크론 잡으로도 잘 작동합니다. 서비스를 생성할 때 적절한 서비스 유형을 선택합니다. 크론 잡은 `PORT`에 바인드할 필요가 없습니다. 백그라운드 워커는 내부 또는 공개 HTTP 엔드포인트도 노출하는 경우에만 바인드해야 합니다.

## 일반적인 문제

**빌드가 `no Go files in .`으로 실패함**
빌드 명령이 잘못된 디렉터리에서 실행되고 있습니다. **루트 디렉터리**를 `go.mod`가 있는 폴더로 설정하거나 빌드 명령을 올바른 패키지 경로(예: `go build -o server ./cmd/myapp`)를 대상으로 업데이트합니다.

**배포 후 서비스가 즉시 재시작됨**
앱이 환경에서 `PORT`를 읽고 해당 포트에서 수신하는지 확인합니다. 즉시 종료되는 프로세스(예: 포트가 이미 사용 중이거나 플래그가 잘못된 경우)는 Render가 반복적으로 재시작합니다.

**빌드 중 모듈 다운로드 오류**
모든 의존성이 `go.sum`에 나열되어 있고 커밋되어 있는지 확인합니다. 비공개 모듈 프록시를 사용하는 경우 서비스의 환경 변수로 `GOPROXY`와 필요한 자격 증명을 설정합니다.
