# Render 工作流

Render 工作流是每次部署服务时运行的完整步骤序列——从触发变更的那一刻，到流量切换到新版本的那一刻。了解每个阶段可以帮助你更快地发布、更早地发现问题，并在出现故障时充满信心地恢复。

## 如何触发部署

每次部署都从触发器开始。Render 支持三种方式：

| 触发器 | 运行时机 |
|---|---|
| **自动部署** | 每次推送到你配置的分支时自动运行（默认启用）。|
| **手动部署** | 通过控制台（**手动部署 → 部署最新提交**）或 Render API 按需触发。|
| **部署钩子** | 通过你从服务设置中生成的经过身份验证的 Webhook URL 按需触发。|

在 **设置 → 构建与部署 → 自动部署** 中配置自动部署。关闭它可以让你在自己的 CI 流水线或审批步骤后再触发部署——仅在检查通过后才调用部署钩子或 API。

### 部署钩子

部署钩子是一个唯一的 HTTPS URL。向该 URL 发送 `POST` 请求即可启动对你配置分支上最新提交的部署。使用钩子可以将 Render 部署集成到任何 CI 系统、脚本或外部服务中，而无需授予其完整的控制台访问权限。

从 **设置 → 构建与部署 → 部署钩子** 生成钩子。请将该 URL 视为 secret——任何拥有该 URL 的人都可以触发部署。

## 构建阶段

触发部署后，Render 在一个全新的构建容器中运行你的 **构建命令**。每次部署的容器是隔离的；上一次构建的文件系统不会保留（模块和包缓存单独存储——见下文[缓存](#缓存)）。

Render 会将构建输出实时流式传输到控制台日志查看器。构建命令的非零退出码会立即导致部署失败。当前版本继续提供服务。

### 各运行时的默认构建命令

| 运行时 | 默认构建命令 |
|---|---|
| Go | `go build -o server .` |
| Node | `npm install; npm run build` |
| Python | `pip install -r requirements.txt` |
| Ruby | `bundle install` |
| Docker | _（从 Dockerfile 构建镜像）_ |

在 **设置 → 构建与部署 → 构建命令** 中覆盖任何默认值。

### 缓存

Render 在构建之间缓存语言包管理器（Go 模块缓存、`node_modules`、pip wheel 缓存等）。缓存按服务和分支进行索引。添加或删除依赖项的提交只会使已更改的包失效——其余缓存保持热态。

要强制执行干净构建，请从控制台选择 **手动部署 → 清除构建缓存并部署**。

## 预部署命令

**预部署命令**在构建成功*之后*、流量切换到新版本*之前*，在一个短暂的容器中运行。将此步骤用于必须与部署保持原子性的任务：

- 数据库迁移
- 缓存预热
- Schema 验证

在 **设置 → 构建与部署 → 预部署命令** 中设置命令。如果命令以非零代码退出，Render 会取消部署并保留前一版本。不会中断任何流量。

> **保持预部署命令幂等。** 如果部署在部分失败后被重试，预部署命令会再次运行。迁移和其他改变状态的操作应当可以安全地运行多次。

## 健康检查

新实例启动后，Render 会等待其通过健康检查，然后再将流量路由到它。对于 Web 服务，Render 会向你配置的 **健康检查路径**（默认：`/`）发送 HTTP GET 请求。服务必须在超时窗口内返回 `2xx` 响应。

在 **设置 → 健康与告警 → 健康检查路径** 中配置路径。一个专用端点——`/healthz` 或 `/ping`——用于检查关键依赖项（数据库连接、必需配置），能为你提供最可靠的信号。

如果健康检查始终未通过：

- 部署被标记为失败。
- 前一版本继续提供服务。
- 新实例被停止。

## 部署生命周期一览

```
收到触发器
  └─ 构建命令运行
       └─ [构建失败] → 部署取消，前一版本保留
       └─ 构建成功
            └─ 预部署命令运行（如已配置）
                 └─ [预部署失败] → 部署取消，前一版本保留
                 └─ 预部署成功
                      └─ 新实例启动，健康检查开始
                           └─ [健康检查失败] → 部署失败，前一版本保留
                           └─ 健康检查通过
                                └─ 流量切换到新版本
                                     └─ 前一实例排空并停止
```

## 零停机部署

Render 对付费计划执行零停机部署。新版本必须通过健康检查，旧版本才会收到停止信号。在重叠窗口期间，两个实例同时运行——请设计你的应用，使两个版本能够安全共存：

- 使数据库 schema 更改与你代码的前一版本向后兼容。
- 不要在同一次部署中既删除列或重命名字段，又添加期望新 schema 的代码。
- 使用预部署命令在新代码开始提供服务*之前*运行迁移。

## 回滚

要回滚到之前的部署，请在控制台打开服务，进入 **事件** 选项卡，找到你想恢复的部署，然后选择 **回滚到此部署**。Render 会为回滚的版本运行相同的工作流——包括预部署命令。

由于回滚会重放预部署命令，请确保迁移是可逆的，或者确认将代码回滚到旧版本对当前数据库 schema 是安全的。

## Pull Request 预览

为服务启用 **Pull Request 预览** 后，Render 会为每个开放的 Pull Request 自动创建该服务的临时副本。预览使用与主服务相同的构建命令、环境变量（减去你明确排除的变量）和设置。

预览在 PR 开启时创建，在 PR 关闭或合并时销毁。预览 URL 的链接会直接发布到 Pull Request 中。

在 **设置 → Pull Request 预览** 中启用预览。你可以选择继承所有环境变量，或为预览环境单独配置一套变量。

> **预览环境共享资源。** 如果你的服务连接了数据库，预览将连接到同一数据库，除非你为预览覆盖了 `DATABASE_URL`。使用单独的数据库或 Schema 进行隔离的预览测试。

## 通知与告警

Render 可以在部署成功、失败或回滚时通知你。在 **设置 → 通知** 中配置通知目标：

- **电子邮件** — 发送给服务所有者或已配置的地址。
- **Slack** — 通过 Webhook 将部署事件发布到频道。
- **PagerDuty / 通用 Webhook** — 将事件转发到任意端点。

健康检查失败也会触发告警。在 **设置 → 健康与告警** 中设置告警阈值。

## 环境晋级模式

对于需要经历 staging 和生产环境的服务，一种常见模式是：

1. **Staging 服务** — 每次推送到 `main` 时自动部署。部署钩子已禁用。
2. **生产服务** — 自动部署已关闭。仅在 staging 通过你的验收标准后才调用部署钩子。

这样在 staging 和生产之间设置了一道人工或自动化的关卡，同时两个环境都使用相同的 Render 工作流基础设施。

## 常见问题

**构建成功但部署始终未上线**
健康检查可能正在失败。检查你的应用是否绑定到 `PORT` 环境变量，并在超时前在你配置的健康检查路径上响应 `2xx`。

**即使迁移已应用，预部署命令每次都会运行**
这是预期行为——请将你的迁移工具设计为当 schema 已是最新时是无操作的（大多数迁移框架默认如此）。

**自动部署在我未配置的分支上触发**
在 **设置 → 构建与部署 → 分支** 中验证分支名称。在你的仓库中重命名分支不会自动更新 Render 中的设置。

**新部署后旧版本继续提供流量**
在免费版服务上，Render 不执行零停机部署。当旧实例停止而新实例启动时，会有一段短暂的时间两者都不提供服务。
